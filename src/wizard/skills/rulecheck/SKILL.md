# Rulecheck

## Role

You are a **code-quality enforcer**. Your job is to delegate a full CLAUDE.md compliance scan to an isolated subagent, wait for results, and relay them verbatim to the user. You do NOT scan or fix files yourself.

---

## Two-Agent Pattern

| Layer | Who | What |
|---|---|---|
| Orchestrator | You (main agent) | Accept focus input, dispatch subagent, relay results |
| Worker | Subagent in isolated worktree | Scan, fix Tier 1 violations, test, open PR |

The orchestrator never touches source files. The subagent never communicates with the user directly — it returns a structured result that you relay verbatim.

---

## Orchestrator Protocol

### Before Dispatching

1. If the user provided a focus area (e.g. "focus on SLAP violations"), record it as `FOCUS=<area>`. If no focus given, set `FOCUS=all`.
2. Confirm the repo is on a clean branch (`git status --short`). If there are staged changes, tell the user and abort.

### Dispatching the Subagent

Use `isolation: worktree` so the subagent works on its own branch. Pass verbatim:

```
You are a CLAUDE.md compliance worker in an isolated git worktree.
FOCUS: <FOCUS value>
RULES REFERENCE: src/wizard/skills/rulecheck/rules-guide.md
Follow the Worker Protocol in that file exactly.
```

### After the Subagent Returns

Relay the subagent's full structured result block to the user without editing or summarising.

---

## Worker Protocol

*This section is read by the subagent. The orchestrator does not execute these steps.*

### Phase 1 — Read Rules

Read `src/wizard/skills/rulecheck/rules-guide.md` in full before scanning.

### Phase 2 — Scan

Scan every `src/wizard/**/*.py` and `tests/**/*.py`. For each violation:
- Record file path, line number, rule name, tier, one-sentence description.
- Do NOT fix anything in this phase.

If `FOCUS` is not `all`, restrict to violations matching the named rule(s) only.

### Phase 3 — Fix Tier 1 Violations

Fix only Tier 1 findings. For each fix:
- Make the minimal change that resolves the violation.
- If a fix would touch more than 3 files or alter a public interface, promote to "recommended" and skip.

Tier 2 violations are documented only — no code changes.

### Phase 4 — Validate

Both must pass before opening a PR:
```bash
uv run pytest -q
ruff check src/
```

If either fails, attempt one fix. If still failing, revert the offending Tier 1 fix (leave as recommended) and re-run. Do not open a PR with a broken test suite.

### Phase 5 — Commit and Open PR

```bash
git add -A
git commit -m "fix(rulecheck): fix Tier 1 CLAUDE.md violations"
gh pr create --title "fix(rulecheck): Tier 1 CLAUDE.md violations" --body "$(cat <<'EOF'
## Summary
Auto-generated by the rulecheck skill.

### Fixes Applied (Tier 1)
<!-- paste Tier 1 rows -->

### Recommended (Tier 2 / Promoted)
<!-- paste Tier 2 + promoted rows -->

### Validation
- pytest: passed
- ruff: passed
EOF
)"
```

### Phase 6 — Return Structured Result

```
## Rulecheck Result

### Violations Found
| # | File | Line | Rule | Tier | Description |

### Fixes Applied (Tier 1)
| # | File | Rule | Change |

### Recommended (Tier 2 / Promoted)
| # | File | Rule | Recommendation |

### Validation
- pytest: [passed / failed]
- ruff: [passed / N errors]

### PR
[URL or "not opened — validation failed"]
```
