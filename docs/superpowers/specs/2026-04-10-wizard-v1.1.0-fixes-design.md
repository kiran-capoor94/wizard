# Wizard v1.1.0 Fixes + Notion Sync Design

**Status:** Approved  
**Date:** 2026-04-10  
**Scope:** Fix remaining code review issues, add Notion sync, add ingest_meeting + create_task tools, remove dead Krisp code

---

## 1. Problem

The v1.1.0 implementation has:
- 6 mechanical code quality issues from the final review
- No Notion read-sync (spec mentioned `_sync_notion` but it was never built)
- Dead `KrispClient` code (Krisp has no REST API — only MCP/webhooks)
- No way for Claude Code to push Krisp meeting data into Wizard
- No way to create tasks from meeting extraction

## 2. Data Flow (Revised)

```
Krisp MCP ──(Claude Code)──> ingest_meeting ──> Wizard DB ──> Notion Meeting Notes DB
Notion Tasks DB ──(sync)──> Wizard DB
Notion Meeting Notes DB ──(sync)──> Wizard DB
Jira ──(sync)──> Wizard DB
Wizard DB ──(write-back)──> Jira (status)
Wizard DB ──(write-back)──> Notion (tasks, meetings, summaries)
```

Claude Code is the orchestrator. It pulls from Krisp MCP and pushes into Wizard. Wizard syncs bidirectionally with Notion and Jira.

## 3. Notion Workspace Schema

### Tasks DB (`notion.tasks_db_id`)
| Property | Type | Maps to |
|----------|------|---------|
| Task | title | `Task.name` |
| Status | status (Not started/In progress/Blocked/Done/Archive) | `Task.status` |
| Priority | select (High/Medium/Low) | `Task.priority` |
| Jira | url | Extract issue key → `Task.source_id`, full URL → `Task.source_url` |
| Due date | date | `Task.due_date` |
| Category | select (Dev/Ops/Planning) | Ignored (Notion-specific, no local mapping) |
| Product | select (Halo/Universe/SISU IQ/Cross-cutting) | Ignored |
| Source | text | Ignored |
| Meeting Note | relation | Ignored (Wizard has its own meeting-task linking) |
| Page ID | — | `Task.notion_id` |

### Meeting Notes DB (`notion.meetings_db_id`)
| Property | Type | Maps to |
|----------|------|---------|
| Meeting name | title | `Meeting.title` |
| Category | multi_select (Planning/Standup/Presentation/Retro/Customer call) | First matching → `Meeting.category`, fallback GENERAL |
| Summary | text | Not overwritten if local summary exists |
| Krisp URL | url | Extract meeting ID → `Meeting.source_id`, full URL → `Meeting.source_url` |
| Date | date | Stored as `Meeting.created_at` if present |
| Product | multi_select | Ignored |
| Tasks | relation | Ignored (Wizard manages its own links) |
| Page ID | — | `Meeting.notion_id` |

## 4. Status/Priority Mapping

### Notion Status → Local TaskStatus
| Notion | Local |
|--------|-------|
| Not started | `todo` |
| In progress | `in_progress` |
| Blocked | `blocked` |
| Done | `done` |
| Archive | `archived` |

### Local TaskStatus → Notion Status (write-back)
Reverse of the above.

### Local TaskStatus → Jira Transition Names (write-back)
| Local | Jira |
|-------|------|
| `todo` | "To Do" |
| `in_progress` | "In Progress" |
| `blocked` | "Blocked" |
| `done` | "Done" |

### Meeting Category Mapping (Notion → Local)
| Notion | Local |
|--------|-------|
| Standup | `standup` |
| Planning | `planning` |
| Retro | `retro` |
| Presentation | `general` (no local equivalent) |
| Customer call | `general` (no local equivalent) |
| (unknown) | `general` |

First matching category wins when multi_select has multiple values.

Priority: 1:1 mapping (High→high, Medium→medium, Low→low).

## 6. New Tools

### `ingest_meeting`
```python
def ingest_meeting(
    title: str,
    content: str,
    source_id: str | None = None,    # Krisp meeting ID
    source_url: str | None = None,   # Krisp URL
    category: str = "general",
) -> dict:
```

**Behaviour:**
1. Scrub title and content via SecurityService
2. Check for existing meeting by `source_id` (dedup)
3. Create or update Meeting record
4. Write back to Notion Meeting Notes DB immediately (fail-safe)
5. Return `IngestMeetingResponse(meeting_id=..., already_existed=...)`

### `create_task`
```python
def create_task(
    name: str,
    priority: str = "medium",
    category: str = "issue",
    source_id: str | None = None,     # Jira key if known
    source_url: str | None = None,    # Jira URL if known
    meeting_id: int | None = None,    # Link to meeting if from extraction
) -> dict:
```

**Behaviour:**
1. Scrub name via SecurityService
2. Create Task record with status=TODO
3. If `meeting_id` provided, create MeetingTasks link
4. Write back to Notion Tasks DB immediately
5. Return `CreateTaskResponse(task_id=..., notion_write_back_succeeded=...)`

### New Response Schemas
```python
class IngestMeetingResponse(BaseModel):

    already_existed: bool

class CreateTaskResponse(BaseModel):
    task_id: int
    notion_write_back_succeeded: bool
```

## 8. SyncService Changes

### sync_all (revised)
```python
def sync_all(self, db: Session) -> None:
    for source, fn in [
        ("jira", self._sync_jira),
        ("notion_tasks", self._sync_notion_tasks),
        ("notion_meetings", self._sync_notion_meetings),
    ]:
        try:
            fn(db)
        except Exception as e:
            logger.warning("Sync failed for %s: %s", source, e)
```

Order: Jira first, then Notion. Notion goes second so it can find tasks already created by Jira sync and set `notion_id` on them rather than creating duplicates.

### _sync_notion_tasks(db)
1. Call `notion.fetch_tasks(tasks_db_id)`
2. For each task:
   - If Jira URL present → extract issue key, look up existing task by `source_id`
   - Else look up by `notion_id`
   - If found: update name, priority, due_date (external wins). Set `notion_id` if not set.
   - If not found: create new Task with `notion_id` set
3. Scrub all text fields
4. Commit

### _sync_notion_meetings(db)
1. Call `notion.fetch_meetings(meetings_db_id)`
2. For each meeting:
   - If Krisp URL present → extract ID, look up existing by `source_id`
   - Else look up by `notion_id`
   - If found: update title, category (external wins). Don't overwrite summary.
   - If not found: create new Meeting with `notion_id` set
3. Scrub all text fields
4. Commit

### Dedup Logic
A task with both a Jira key and a Notion page ID should be **one record** with both `source_id` (Jira) and `notion_id` (Notion) set. The lookup order is:
1. Try `source_id` match first (most authoritative)
2. Try `notion_id` match second
3. Create new if neither matches

Same pattern for meetings (Krisp source_id → Notion notion_id).

## 10. Cleanup Fixes

### 1. Session management
All tool functions use `try/finally` to ensure `db.close()`:
```python
db = _get_db()
try:
    # ... tool logic ...
finally:
    db.close()
```

### 2. `_get_deps()` caching
Lazy singleton — create deps once, reuse:
```python
_cached_deps = None

def _get_deps():
    global _cached_deps
    if _cached_deps is None:
        _cached_deps = _build_deps()
    return _cached_deps
```

### 3. httpx timeout
Add `timeout=httpx.Timeout(10.0)` to all Jira httpx calls. NotionClient uses `notion-sdk-py` which handles its own timeouts.

### 4. requires-python
Change `pyproject.toml` from `>=3.14` to `>=3.12`.

### 5. Unused imports
Remove `Optional` from `integrations.py`, any other unused imports flagged by linters.
