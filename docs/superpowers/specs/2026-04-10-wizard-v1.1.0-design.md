# Wizard v1.1.0 — Design

**Status:** Approved
**Date:** 2026-04-10
**Target:** Local-first Memory Layer for AI Agents (Claude Code)

---

## 1. Purpose

Wizard is the integration and memory layer between Claude Code and external systems (Jira, Notion, Krisp). Claude Code is the reasoning engine — Wizard remembers. It syncs external data, scrubs PII at every boundary, persists Claude Code's reasoning as structured notes, and surfaces prior context across sessions so reasoning compounds over time.

---

## 2. Module Structure

```
src/
  config.py          — Settings with nested Jira/Notion/Krisp/Scrubbing blocks
  database.py        — SQLModel engine + get_session context manager (unchanged)
  models.py          — aligned schema (see Section 3)
  schemas.py         — typed Pydantic response models for all MCP tools
  security.py        — RegexSecurityService + ScrubResult; module-level singleton
  repositories.py    — NoteRepository (dual-lookup: task_id + source_id)
  integrations.py    — JiraClient, KrispClient, NotionClient
  services.py        — SyncService (ingest + upsert), WriteBackService
  tools.py           — @mcp.tool() functions; thin wrappers only
  server.py          — FastMCP init, imports tools

tests/
  test_models.py     — updated for schema changes; source_id uniqueness
  test_security.py   — scrub patterns, allowlist, idempotency
  test_repositories.py — NoteRepository dual-lookup behaviour
  test_tools.py      — each MCP tool against in-memory DB; typed response assertions
```

**Data flows:**

- **Ingestion (session_start):** `tools → SyncService → [JiraClient / KrispClient / NotionClient] → SecurityService.scrub() → repositories / db.add()`
- **Save note (inbound from Claude Code):** `tools → SecurityService.scrub() → NoteRepository.save()`
- **Write-back (update_task_status, save_meeting_summary, session_end):** `tools → WriteBackService → [JiraClient / NotionClient]`

Write-back failure logs and continues. Local state always persists regardless of write-back outcome.

## 3. Schema

### Changes from current `src/models.py`

| Model            | Change                                                                                                                                                            |
| ---------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
| `Task`           | Add `notion_id: Optional[str]` (indexed); `source_id` → `unique=True`; remove `session_id` FK + `session` relationship; keep `due_date`, `category: TaskCategory` |
| `Meeting`        | Add `title: str`, `notion_id: Optional[str]` (indexed); `source_id` → `unique=True`; `MeetingCategory` gains `GENERAL`, loses `PRESENTATION` and `TRAINING`       |
| `Note`           | Add `meeting_id: Optional[int]` FK; `NoteType` gains `SESSION_SUMMARY`; keep `note_type` field name (not `type`); keep `source_type`                              |
| `WizardSession`  | Remove `metadata_dump`; remove `tasks` relationship; `__tablename__` → `"wizardsession"`                                                                          |
| `TimestampMixin` | No change                                                                                                                                                         |

### `source_id` uniqueness contract

- `Task.source_id` — `unique=True`. One task per Jira ticket.
- `Meeting.source_id` — `unique=True`. One meeting per Krisp transcript.
- `Note.source_id` — `index=True` only. Multiple notes may reference the same source.

## 4. Security

**File:** `src/security.py`

`SecurityService` is a module-level singleton (`security_service = SecurityService()`). Imported directly in `tools.py` and `services.py`. Nothing enters or leaves Wizard storage without passing through `scrub()`.

**Two scrub points:**

- **Outbound (ingestion):** External data scrubbed by `SyncService` before `db.add()`. `wizard.db` always contains clean data.
- **Inbound (agent output):** Claude Code content scrubbed in `save_note`, `save_meeting_summary`, and `session_end` before write.

**Redaction:** Indexed semantic stubs — `[EMAIL_1]`, `[NHS_ID_1]`, etc. Index preserves relational context across a document. No mapping is stored. Stubs are permanent and irreversible.

**Patterns:** NHS_ID, NI_NUMBER, EMAIL, PHONE, POSTCODE, SECRET (Bearer tokens, sk- keys).

**Allowlist:** Defined inline in `config.json` under `scrubbing.allowlist` as a list of strings (regex or plain). Applied before pattern matching. Empty list = no allowlist entries.

**`ScrubResult`:**

```python
@dataclass
class ScrubResult:
    clean: str
    stubs_applied: Dict[str, str]  # original → stub; for audit only, never stored
    was_modified: bool
```

**Known limitation:** No named-entity recognition — person names are not detected. Acceptable for v1.1.0; structured identifiers (NHS IDs, NI numbers, emails) are the primary PII risk in SISU Health context.

## 5. Response Models

**File:** `src/schemas.py`

Typed Pydantic models. FastMCP serialises them automatically. Error paths raise exceptions — FastMCP converts to MCP errors. No `{"status": "error"}` dicts.

Responses are opinionated: they surface what Claude Code needs to reason on next without requiring additional tool calls to determine state.

```python
class TaskContext(BaseModel):
    id: int
    name: str
    status: TaskStatus
    priority: TaskPriority
    category: TaskCategory
    due_date: Optional[datetime]
    source_id: Optional[str]
    source_url: Optional[str]
    last_note_type: Optional[NoteType]      # type of most recent note on this task
    last_note_preview: Optional[str]        # first 300 chars of most recent note
    last_worked_at: Optional[datetime]      # created_at of most recent note

class MeetingContext(BaseModel):
    id: int
    title: str
    category: MeetingCategory
    created_at: datetime
    has_summary: bool                       # False → Claude Code should call get_meeting

class NoteDetail(BaseModel):
    id: int
    note_type: NoteType
    content: str
    created_at: datetime
    source_id: Optional[str]

class SessionStartResponse(BaseModel):
    session_id: int
    open_tasks: list[TaskContext]           # HIGH priority first, then by last_worked_at desc
    blocked_tasks: list[TaskContext]        # surfaced separately — different handling required
    unsummarised_meetings: list[MeetingContext]  # has_summary=False; call get_meeting on these

class TaskStartResponse(BaseModel):
    task: TaskContext
    compounding: bool                       # True = prior notes exist
    notes_by_type: dict[str, int]          # {"investigation": 3, "decision": 1}
    prior_notes: list[NoteDetail]          # full content, newest first

class SaveNoteResponse(BaseModel):
    note_id: int

class UpdateTaskStatusResponse(BaseModel):
    task_id: int
    new_status: TaskStatus
    write_back_succeeded: bool             # False = Jira/Notion out of sync; Claude Code can warn

class GetMeetingResponse(BaseModel):
    meeting_id: int
    title: str
    category: MeetingCategory
    content: str                            # full transcript
    already_summarised: bool
    existing_summary: Optional[str]
    open_tasks: list[TaskContext]          # candidate action items for Claude Code to match against transcript

class SaveMeetingSummaryResponse(BaseModel):
    note_id: int
    linked_task_ids: list[int]

class SessionEndResponse(BaseModel):
    note_id: int
```

## 6. Internal Layers

### `NoteRepository` (`src/repositories.py`)

Single responsibility: DB access for `Note`. Dual-lookup: by `task_id` and by `source_id` — notes are linked to both the local task record and the external source identifier.

```python
class NoteRepository:
    def save(self, db: Session, note: Note) -> Note: ...
    def get_for_task(self, db: Session, task_id: int, source_id: Optional[str]) -> list[Note]:
        # Returns notes where note.task_id == task_id OR note.source_id == source_id, ordered created_at desc.
        # The OR exists because notes imported from Notion may carry a source_id matching a Jira key
        # but no task_id (they were created before the task was synced locally). Both must be found
        # for the compounding loop to surface prior context correctly.
```

### `SyncService` (`src/services.py`)

Orchestrates ingestion from all three sources. Each source is wrapped in `try/except` — partial failure logs and continues. Upsert contract: SELECT by `source_id` first. External source wins on all matched fields except `status` (not overwritten if locally modified) and `notion_id` (Wizard-owned, never overwritten).

```python
class SyncService:
    def sync_all(self, db: Session) -> None:
        self._sync_jira(db)    # → Task upserts
        self._sync_krisp(db)   # → Meeting upserts
        self._sync_notion(db)  # → Note upserts
```

**Upsert field rules (explicit):**

- `name`, `source_url`, `priority`, `category`, `title`, `content` — external source wins, always overwritten
- `status` — NOT overwritten. Local changes via `update_task_status` are preserved.
- `notion_id` — NOT overwritten. Wizard-owned; set on first Notion write-back, never reset by sync.

**Missing credentials:** If a client's token is absent, `SyncService` skips that source and logs a warning. `session_start` still succeeds with whatever data is available locally.

### `WriteBackService` (`src/services.py`)

Handles push to Jira/Notion after local state changes. Failure is non-fatal — local state is always committed before write-back is attempted. Write-back outcome is returned to the tool as a `bool`, surfaced in the typed response (`write_back_succeeded`), so Claude Code can warn the user when Jira/Notion is out of sync. No automatic recovery in v1.1.0 — manual reconciliation via the next `update_task_status` call.

```python
class WriteBackService:
    def push_task_status(self, task: Task) -> bool: ...          # returns success flag
    def push_meeting_summary(self, meeting: Meeting) -> bool: ...
    def push_session_summary(self, session: WizardSession) -> bool: ...
```

### Clients (`src/integrations.py`)

`JiraClient`, `KrispClient`, `NotionClient` — each takes credentials from `Settings`. Missing credentials → client raises `ConfigurationError` at init time; `SyncService` catches per-client and logs. Clients are not protocol-unified — their payloads differ too much to share an interface in v1.1.0.

## 7. Configuration

All configuration — including credentials — lives in `~/.wizard/config.json`. This is a personal, single-user tool; the config file sits outside the repo in the user's home directory and is not committed to git. No env var overrides for secrets.

`src/config.py` extended with nested settings blocks:

```python
class JiraSettings(BaseModel):
    base_url: str
    project_key: str
    token: str

class NotionSettings(BaseModel):
    daily_page_id: str
    token: str

class KrispSettings(BaseModel):
    api_base_url: str
    token: str

class ScrubbingSettings(BaseModel):
    enabled: bool = True
    allowlist: list[str] = []   # regex or plain strings; applied before PII pattern matching

class Settings(BaseSettings):
    ...
    jira: JiraSettings
    notion: NotionSettings
    krisp: KrispSettings
    scrubbing: ScrubbingSettings = ScrubbingSettings()
```

## 8. Migration Strategy

**Squash:** The existing initial migration (`f9c1f7acf832_initial.py`) covers an incomplete schema. Since there is no production DB, the approach is:

1. Delete `alembic/versions/f9c1f7acf832_initial.py`
2. Wipe `wizard.db`
3. Align `src/models.py` to target schema
4. Run `alembic revision --autogenerate -m "initial"` — produces a single migration = current truth
5. Run `alembic upgrade head` — clean DB from scratch

One migration in history. No layering.

## 10. End-to-End Proof (Compounding Loop)

1. `session_start` → Task A in `open_tasks`
2. `task_start(task_id=A)` → `compounding=False`, empty `prior_notes`
3. Claude Code calls `save_note(task_id=A, note_type="investigation", ...)`
4. `update_task_status(task_id=A, status="in_progress")`
5. `session_end(summary=...)`
6. New session: `session_start` → Task A still in `open_tasks`, `last_note_type="investigation"`
7. `task_start(task_id=A)` → `compounding=True`, `prior_notes` contains note from step 3

**Proved when step 7 returns `compounding=True` with a non-empty `prior_notes`.**

**Meeting proof:**

1. Krisp sync → Meeting M in DB, `has_summary=False` in `session_start` response
2. `get_meeting(meeting_id=M)` → transcript + `already_summarised=False` + `open_tasks`
3. Claude Code calls `save_meeting_summary(meeting_id=M, task_ids=[A])`
4. Verify: Meeting M has summary, Task A linked to Meeting M, DOCS Note exists with `meeting_id=M`

## 11. Non-Goals (v1.1.0)

- Code intelligence (AST, TreeSitter, embeddings)
- Vector / semantic search
- Reversible PII scrubbing — stubs are permanent, no mapping stored
- Multi-user support
- Cloud sync
- `blast_radius` tool (deferred to v1.2.0)
